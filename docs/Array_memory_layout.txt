多维数组的内存布局
================================================================================

一、C 连续存储（行主序，Row-Major）
================================================================================

  C/C++ 和 PyTorch 默认均使用行主序：最右边的维度在内存中连续排列。

  以 shape [2, 3, 4] 为例，共 24 个元素，内存地址从低到高：

    [0,0,0] [0,0,1] [0,0,2] [0,0,3]   ← 最右维（z=4）连续
    [0,1,0] [0,1,1] [0,1,2] [0,1,3]
    [0,2,0] [0,2,1] [0,2,2] [0,2,3]
    [1,0,0] [1,0,1] [1,0,2] [1,0,3]
    [1,1,0] [1,1,1] [1,1,2] [1,1,3]
    [1,2,0] [1,2,1] [1,2,2] [1,2,3]

  元素 [i,j,k] 的内存偏移：
    offset = i*(y*z) + j*z + k
           = i*12   + j*4  + k      （y=3, z=4）


二、[x,y,z] 与 [x*y,z] 内存完全相同
================================================================================

  两者总元素数相同（x*y*z），且在行主序下字节顺序一致。

  [x,y,z] 中元素 [i,j,k] 的偏移：
    offset = i*(y*z) + j*z + k

  [x*y,z] 中元素 [i*y+j, k] 的偏移：
    offset = (i*y+j)*z + k = i*(y*z) + j*z + k   ← 完全相同

  结论：两种 shape 是同一块内存的不同解读方式，不需要任何数据拷贝。

  示例（shape [2,3,4] vs [6,4]）：

    [2,3,4] 视角：        [6,4] 视角：          内存地址：
    [0,0,:] → 4个元素     [0,:] → 4个元素       addr+0
    [0,1,:] → 4个元素     [1,:] → 4个元素       addr+16
    [0,2,:] → 4个元素     [2,:] → 4个元素       addr+32
    [1,0,:] → 4个元素     [3,:] → 4个元素       addr+48
    [1,1,:] → 4个元素     [4,:] → 4个元素       addr+64
    [1,2,:] → 4个元素     [5,:] → 4个元素       addr+80


三、stride（步长）是理解多维数组的核心
================================================================================

  shape 只描述"有几行几列"，stride 描述"相邻元素在内存中相差多少个元素"。

  shape [2,3,4] 的 strides：
    dim 0（x）: stride = 3*4 = 12   → 相邻行首元素间隔 12 个元素
    dim 1（y）: stride = 4          → 相邻列首元素间隔 4 个元素
    dim 2（z）: stride = 1          → 相邻元素紧挨着

  shape [6,4] 的 strides：
    dim 0: stride = 4
    dim 1: stride = 1

  只要 stride 不变，shape 怎么分组解读都行。
  PyTorch 的 view() 和 reshape() 正是通过修改 shape 和 stride 元数据实现的，
  不移动任何数据。


四、reshape / view 的本质
================================================================================

  view()   : 要求内存连续（is_contiguous() == true），直接改 shape/stride 元数据
  reshape(): 内存连续时等价于 view()；不连续时先 contiguous()（拷贝一次）再 view()

    x = torch.randn(2, 3, 4)          # shape [2,3,4]，连续
    y = x.view(6, 4)                  # shape [6,4]，共享同一块内存，无拷贝
    z = x.reshape(6, 4)               # 同上，连续时等价

    x_t = x.transpose(0, 1)           # shape [3,2,4]，内存不连续（stride 顺序打乱）
    x_t.view(6, 4)                    # 报错：内存不连续
    x_t.reshape(6, 4)                 # 先拷贝成连续，再 view，有数据移动

  判断是否连续（C-contiguous）：
    从最后一维往前，stride[i] == stride[i+1] * size[i+1]
    shape [2,3,4]：stride[1]=4 == stride[2]*size[2]=1*4 ✓
                   stride[0]=12 == stride[1]*size[1]=4*3 ✓  → 连续


五、本项目中的应用
================================================================================

  softmax kernel 接受 [h, w] 的二维视图，对每行做 softmax。
  三维输入 [batch_size, seq_len, vocab_size] 内存连续，可直接展平前两维：

    int64_t w = x.size(-1);        // vocab_size，最后一维
    int64_t h = x.numel() / w;     // batch_size * seq_len，其余维度之积

  内存视角：

    [batch_size, seq_len, vocab_size] 连续存储时：
      row 0  : x[0,0,:]   → vocab_size 个元素
      row 1  : x[0,1,:]   → vocab_size 个元素
      ...
      row h-1: x[B-1,S-1,:] → vocab_size 个元素

    等价于 [h, w] = [batch*seq, vocab]，每行是一个独立的 softmax 计算单元。

  kernel 只看到线性地址 + w + h，不感知维度数，三维和二维无差别处理。

  LLM 典型参数（LLaMA-3）：
    shape    : [batch_size, seq_len, 128256]
    w        : 128256（vocab_size）
    每行内存 : 128256 × 4 bytes = 512 KB（float32）
    h        : batch_size × seq_len


六、[x,y,z] 与 [x*y,z] 不等价的情况
================================================================================

  内存不连续时，两者不能直接互换：

    x = torch.randn(3, 2, 4)
    x_t = x.transpose(0, 1)    # shape [2,3,4]，strides=[4,12,1]（非标准顺序）

    # 此时 [2,3,4] 与 [6,4] 内存布局不同：
    # [2,3,4] 的 stride[0]=4，相邻"行"间隔 4 个元素（不是 3*4=12）
    # 展平成 [6,4] 需要 stride[0]=4，但元素实际上不是按 [6,4] 顺序排列的

    x_t.view(6, 4)              # RuntimeError: 内存不连续，无法 view

  只有满足 C-contiguous 条件（stride 从右到左递增且连续），
  [x,y,z] 才能安全地被视为 [x*y,z]。
